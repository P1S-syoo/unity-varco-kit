# github-commit-push — 커밋·푸시 절차

커밋은 "스냅샷 저장"이 아니라 "되돌릴 수 있는 작업 단위 기록"이다. 아래 절차를 순서대로 따른다.

## 1. 상태 파악

```
git status --short
git log --oneline -5
```

- 변경 파일 전체를 훑고, `git log`로 이 저장소의 기존 커밋 메시지 스타일을 확인한다 (스타일이 다르면 기존 스타일이 우선).
- 의도하지 않은 파일(생성물, 임시 파일)이 보이면 커밋하지 말고 .gitignore 보강을 먼저 제안한다.

## 2. 사전 점검 (커밋 전 필수)

순서대로 확인하고, 하나라도 걸리면 커밋을 멈추고 사용자에게 보고한다:

- **시크릿 스캔**: 스테이징 대상에서 API 키/토큰 패턴(`api[_-]?key`, `token`, `secret`, `password`, `Bearer ` 등)을 grep. 발견 시 절대 커밋하지 않는다 — 한 번 푸시된 시크릿은 히스토리에 영구히 남는다.
- **대용량 파일**: 50MB 초과 파일은 경고, 100MB 초과는 GitHub가 거부하므로 Git LFS 또는 .gitignore 처리를 제안한다.
- **유니티 .meta 짝**: 새 에셋/폴더에 .meta가 함께 스테이징되는지, 삭제된 에셋의 .meta도 함께 삭제되는지 확인한다. 짝이 어긋나면 다른 PC에서 임포트가 깨진다.
- **생성물 혼입**: `Library/`, `Temp/`, `obj/`, `.csproj`, `.sln` 등이 스테이징에 보이면 .gitignore 누락이다.

## 3. 원자적 커밋 분리

변경을 의미 단위로 나눈다 — "오늘 한 일 전부"를 한 커밋에 넣지 않는다:

- 기능 추가 / 버그 수정 / 리팩토링 / 문서는 각각 별도 커밋.
- 유니티 에셋과 그 .meta, 코드와 그 코드가 참조하는 에셋은 같은 커밋에 묶는다 (한 커밋만 체크아웃해도 동작하도록).
- 서로 무관한 변경이 한 파일에 섞였으면 `git add -p` 대신 가능한 파일 단위로 분리하고, 불가피하면 한 커밋으로 묶되 메시지에 둘 다 적는다.

## 4. 커밋 메시지

**conventional commit 타입 + 한글 요약 한 줄** 형식을 쓴다:

```
feat: 인벤토리 저장/불러오기 추가
fix: 점프 중복 입력으로 2단 점프되던 버그 수정
refactor: PlayerHealth 직접 참조를 이벤트 구독으로 전환
docs: README 셋업 가이드 추가
test: Battery 방전 PlayMode 테스트 추가
chore: .gitignore에 Test Runner 임시 씬 추가
```

- 타입: `feat` / `fix` / `refactor` / `docs` / `test` / `chore` 중 선택.
- 요약은 한글로, "무엇을 왜"가 드러나게 한 줄. 본문이 필요하면 빈 줄 후 2~4줄로 변경 이유를 적는다.
- 리뷰 수정사항을 커밋할 때는 어떤 이슈를 고쳤는지 명시한다 (예: `fix: 리뷰 HIGH — BlockPool 이중 반환 방어 추가`).

## 5. 푸시

```
git remote -v
```

- **원격이 없으면**: `gh repo create <이름> --private --source . --push`를 제안한다 (public 여부는 사용자에게 확인).
- **원격이 있으면**: 푸시 전 `git fetch` 후 behind 상태면 `git pull --rebase`로 먼저 정리한다. 충돌이 나면 멈추고 사용자에게 보고한다.
- main/master에 직접 푸시할지, 브랜치+PR로 갈지는 저장소의 기존 흐름을 따른다 (개인 프로젝트의 단독 작업이면 직접 푸시가 기본).
- 푸시 후 `git log origin/<branch> -1 --oneline`으로 반영을 확인하고 결과를 보고한다.

## 6. 실패 처리

- 인증 실패: `gh auth status` 확인 후 `gh auth login` 안내.
- 푸시 거부(non-fast-forward): 강제 푸시(`--force`)는 사용자가 명시적으로 요청하지 않는 한 절대 쓰지 않는다. `pull --rebase` 경로로 해결한다.
- 훅 실패: 훅을 우회(`--no-verify`)하지 말고 원인을 고친다.
