# code-rules — 코딩 컨벤션 & 작업 절차

유니티 C# 프로젝트에서 코드를 작성할 때 따르는 규칙이다. 핵심은 4가지: **K&R 스타일, try-catch 예외처리, 한글 한 줄 주석, 패턴 결정은 decision tree로**. 그리고 작업 전후로 **codegraph**를 활용한다.

## 작업 순서

코드 작업은 항상 이 순서를 따른다:

1. **구조 파악 (codegraph)** — 수정할 심볼과 영향 범위를 먼저 확인한다. 아래 [codegraph 활용](#codegraph-활용) 참조.
2. **패턴 결정 (decision tree)** — 새 클래스/시스템을 설계하거나 구조 변경이 필요하면, 코드를 쓰기 전에 `references/pattern-decision-tree.md`의 decision tree를 따라가 어떤 디자인패턴을 쓸지(혹은 안 쓸지) 결정하고, 그 결정과 이유를 사용자에게 한 줄로 알린다.
3. **코드 작성** — 아래 코딩 컨벤션을 적용한다.
4. **동기화 (codegraph sync)** — 코드 작업이 끝나면 codegraph 인덱스를 동기화한다.

## 코딩 컨벤션 (요약)

상세 규칙과 예시는 `references/csharp-style.md`를 읽는다. 코드를 실제로 작성하기 전에 반드시 한 번 읽어둔다.

- **K&R 중괄호**: 여는 중괄호는 같은 줄에. C# 기본 관례(Allman)와 다르므로 의식적으로 적용한다.
  ```csharp
  public void Attack(Enemy target) {
      if (target == null) {
          return;
      }
  }
  ```
- **try-catch 예외처리**: 실패 가능한 작업(I/O, 파싱, 리소스 로드, 네트워크, 컴포넌트 접근)을 하는 함수는 try-catch로 감싸고, catch에서 한글 로그를 남긴다. 빈 catch 금지.
- **한글 주석**: 한 줄 이내, 간략하고 알기 쉽게. 의도/이유를 적고 자명한 코드에는 달지 않는다.
- **요청 범위만 수정**: 버그 수정·기능 추가 시 요청과 무관한 방어 코드나 리팩토링을 끼워 넣지 않는다. 개선거리를 발견하면 코드에 넣지 말고 사용자에게 제안만 한다. (단, 수정하는 코드의 스타일을 위 컨벤션에 맞추는 것은 허용)

## 디자인패턴 decision tree

새로운 기능·클래스·시스템을 설계할 때는 `references/pattern-decision-tree.md`를 읽고 트리를 따라간다. 트리의 첫 질문은 "패턴이 정말 필요한가?"다 — 패턴은 목적이 아니라 도구이므로, 단순한 문제는 단순하게 푸는 것이 정답일 수 있다. 트리를 통과한 결과(선택한 패턴 또는 "패턴 불필요")와 근거를 코드 작성 전에 사용자에게 간단히 보고한다.

## codegraph 활용

프로젝트가 커져도 구조를 추적할 수 있도록 codegraph로 심볼을 노드화해 관리한다.

- **작업 전**: `codegraph_context`로 수정 대상 주변 구조를 파악하고, 기존 심볼 변경 시 `codegraph_impact`로 영향 범위를 확인한다. grep으로 심볼을 찾지 말고 `codegraph_search`를 쓴다.
- **작업 후 (필수)**: 코드 작성·수정이 끝나면 항상 codegraph 인덱스를 동기화한다. `codegraph_status`로 인덱스 상태를 확인하고, 변경이 반영되지 않았으면 sync를 실행한다:
  ```
  codegraph sync
  ```
  (CLI가 없거나 실패하면 MCP의 `refresh_index` 도구로 대체한다. 둘 다 불가능하면 사용자에게 알린다.)
- **codegraph가 초기화되지 않은 프로젝트**: `.codegraph/`가 없으면 사용자에게 `codegraph init -i` 실행 여부를 물어본다.
