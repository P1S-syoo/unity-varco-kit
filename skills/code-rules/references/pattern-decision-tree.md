# 디자인패턴 Decision Tree

코드를 설계하기 전에 이 트리를 따라간다. 결과(선택한 패턴 또는 "패턴 불필요")와 근거 한 줄을 사용자에게 보고한 뒤 코드를 작성한다.

근거 출처: GoF 패턴 선택 절차, Refactoring.Guru(코드 냄새), Kerievsky *Refactoring to Patterns*, Fowler YAGNI(*Is Design Dead?*), iluwatar/java-design-patterns(KISS/YAGNI 원칙).

## 0단계: 게이트 질문

**"패턴 없이 단순하게 해결할 수 있는가?"** — 모든 권위 있는 자료(GoF, Fowler, Kerievsky, iluwatar)가 공통으로 두는 첫 관문이다. 패턴은 리팩토링으로 *도달*하는 것이지 처음부터 박아 넣는 것이 아니다.

다음 중 하나라도 해당하면 **패턴 불필요**로 판정하고 단순하게 작성한다:
- 구현(변형)이 현재 1개뿐이고 늘어날 구체적 계획이 없다 (YAGNI)
- 클래스 2~3개 규모의 단순한 기능이다
- 아래 [코드 냄새](#코드-냄새--패턴-신호-매핑)에 해당하는 것이 하나도 없다

"언젠가 필요할 것 같아서"는 패턴 도입 사유가 아니다 (Speculative Generality 냄새). 이 경우 보고는 한 줄이면 된다: `패턴 판단: 불필요 — 변형이 1개뿐이라 단순 구현 (YAGNI)`.

## 1단계: 문제 유형 분기

게이트를 통과했다면(= 패턴이 필요한 신호가 있다면), 문제의 본질을 분류한다:

```
지금 겪는 문제의 본질은?
├── A. 객체를 만드는 방식이 복잡하다 ───────→ 생성(Creational) 트리
├── B. 클래스/모듈 간 관계·결합·인터페이스가 문제다 → 구조(Structural) 트리
├── C. 조건문이 폭증하거나 행동이 상황마다 달라진다 → 행동(Behavioral) 트리
└── D. 성능·수명주기 등 유니티 런타임 특유의 문제다 → 유니티 특화 트리
```

## 2단계: 세부 트리

### A. 생성(Creational) 트리

```
생성할 타입이 런타임에 결정되는가? ──────────────→ Factory Method
연관된 객체들을 일관된 세트로 생성해야 하는가? ──→ Abstract Factory
생성 파라미터가 4개 이상이거나 다단계 구성인가? ─→ Builder
생성 비용이 커서 복사가 더 나은가? ──────────────→ Prototype
전역에서 정확히 1개 인스턴스여야 하는가? ────────→ Singleton (⚠ 아래 경고 참조)
```

### B. 구조(Structural) 트리

```
제어할 수 없는 외부/레거시 인터페이스를 맞춰야 하는가? → Adapter
기존 클래스 수정 없이 기능을 동적으로 덧붙여야 하는가? → Decorator
복잡한 서브시스템에 단순한 진입점이 필요한가? ────────→ Facade
접근 제어·지연 로딩·캐싱이 필요한가? ─────────────────→ Proxy
개별 객체와 그룹(트리)을 동일하게 다뤄야 하는가? ─────→ Composite
추상과 구현이 서로 독립적으로 확장되어야 하는가? ─────→ Bridge
동일 데이터를 가진 객체가 수천 개라 메모리가 문제인가? → Flyweight
```

### C. 행동(Behavioral) 트리

```
알고리즘을 런타임에 교체하거나 큰 switch를 없애야 하는가? → Strategy
한 객체의 변화를 여러 객체에 알려야 하는가? ─────────────→ Observer (C# event/Action)
객체의 행동이 내부 상태에 따라 크게 달라지는가? ─────────→ State (FSM)
작업을 객체화해서 지연 실행·취소·재실행해야 하는가? ─────→ Command
알고리즘 뼈대는 같고 일부 단계만 다른가? ────────────────→ Template Method
여러 컴포넌트 간 통신이 거미줄처럼 얽혔는가? ────────────→ Mediator
요청이 여러 핸들러를 순차 통과해야 하는가? ──────────────→ Chain of Responsibility
구조는 안정적인데 연산 종류만 계속 늘어나는가? ──────────→ Visitor
```

### D. 유니티 특화 트리 (추천 의견)

GoF 트리에 더해, 유니티 게임 개발에서 실제로 가장 자주 쓰이는 결정들이다:

```
총알·이펙트·적처럼 생성/파괴가 잦은 오브젝트인가?
  ──→ Object Pool (Instantiate/Destroy 반복은 GC 스파이크의 주범)

여러 시스템(UI·사운드·퀘스트)이 게임 이벤트에 반응해야 하는가?
  ──→ Observer: C# event/Action 우선, 에디터 노출이 필요하면 UnityEvent,
      씬을 넘는 전역 이벤트면 ScriptableObject 이벤트 채널

캐릭터 AI·애니메이션처럼 상태 전이가 명확한가?
  ──→ State (FSM) — 유니티에서 가장 투자 가치가 높은 패턴

GameManager류 전역 매니저가 필요한가?
  ──→ Singleton은 최소화하고, 데이터/설정 공유는 ScriptableObject,
      서비스 참조는 의존성 주입 또는 Service Locator를 먼저 검토

밸런스 수치·아이템 정의 등 디자이너가 만질 데이터인가?
  ──→ ScriptableObject 데이터 컨테이너 (코드와 데이터 분리)
```

## 코드 냄새 → 패턴 신호 매핑

리팩토링 요청을 받았을 때 이 표로 냄새를 먼저 식별하면 트리 분기가 빨라진다 (Refactoring.Guru / Fowler 기반):

| 코드 냄새 | 패턴 신호 |
|---|---|
| 긴 switch/if-else 사슬 | Strategy, State |
| 유사한 알고리즘이 여러 곳에 중복 | Template Method, Strategy |
| 한 클래스가 여러 이유로 자꾸 변경됨 | Bridge, 클래스 분리 |
| `a.GetB().GetC().GetD()` 메시지 체인 | Facade, Proxy |
| 클래스 간 직접 참조 과다 (과결합) | Mediator, Observer |
| 파라미터 4개 이상 | Builder, 파라미터 객체 |
| "언젠가 쓸" 추상화 (Speculative Generality) | **패턴 제거 신호** — 과잉설계 |

## 안티패턴 경고

패턴을 골랐어도 적용 전에 마지막으로 확인한다:

- **Singleton 남용**: 테스트를 어렵게 하고 숨은 전역 상태를 만든다. 유니티에서도 매니저마다 Singleton을 박지 말고 정말 "정확히 1개 + 전역 접근"이 모두 필요한지 확인한다 (Python Decision Guide, 실무 공통 의견).
- **변형 1개짜리 Factory/Strategy**: 구현이 하나뿐이면 복잡도만 늘어난다. Kerievsky 본인도 "조건문 하나면 충분했는데 Strategy부터 꺼냈다"고 자기고백했다.
- **Decorator 3겹 이상**: 디버깅이 불가능해진다.
- **God Facade**: Facade가 메서드 10개를 넘으면 분할한다.
- **Observer 구독 누수**: 유니티에서는 `OnDestroy`/`OnDisable`에서 반드시 구독 해제한다 — 파괴된 오브젝트 참조로 인한 MissingReferenceException의 단골 원인.
- Fowler의 원칙: **"패턴이 자신의 무게를 감당하지 못하면 제거를 두려워하지 마라."** 패턴은 트로피가 아니라 도구다.
